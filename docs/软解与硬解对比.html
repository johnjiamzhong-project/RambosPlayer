RambosPlayer — 软解 vs 硬解 对比

CPU 软解（现状）与 D3D11VA 硬解（Phase 7 目标）的架构差异与性能对比

软解路径 硬解路径 CPU 执行 GPU 执行 决策分支

🖥 软解路径（现状）

📦 输入：videoPacketQ → AVPacket*
DemuxThread 解复用后的 H.264 压缩包
⬇
avcodec_send_packet()
发送压缩包到解码器
⬇
avcodec_receive_frame()
CPU 通用核心 执行全部解码运算
(熵解码 + 反量化 + 反变换 + 运动补偿 + 去块滤波)
⬇
AVFrame* → 系统内存
帧格式: AV_PIX_FMT_YUV420P
数据已在 CPU 可读的内存中，直接可用
⬇
push → videoFrameQ
推入帧队列，等待 VideoRenderer 消费
⚡ 特点：帧直接在系统内存中，无额外拷贝。
🔴 代价：1080p H.264 解码约占用 40–60% CPU（取决于码率和帧率）。

⚡ 硬解路径（Phase 7 目标）

📦 输入：videoPacketQ → AVPacket*
DemuxThread 解复用后的 H.264 压缩包（同软解）
⬇
🔧 init 阶段：创建 D3D11VA 设备
av_hwdevice_ctx_create(AV_HWDEVICE_TYPE_D3D11VA)
→ 绑定到 AVCodecContext.hw_device_ctx
⬇
avcodec_send_packet()
发送压缩包（CPU 侧，开销极小）
⬇
avcodec_receive_frame()
GPU 专用硅电路 执行解码
(NVDEC 固定功能单元，1650S 支持 H.264/HEVC 完全硬解)
⬇
AVFrame* → 显存 (VRAM)
帧格式: AV_PIX_FMT_D3D11
⚠ 帧在 GPU 显存中，CPU 无法直接读写
⬇ 关键步骤：显存 → 系统内存
⚠ av_hwframe_transfer_data()
将帧从 GPU 显存拷贝到 CPU 系统内存
格式变换: D3D11 → NV12/YUV420P
这是硬解路径唯一新增的开销
⬇
push → videoFrameQ
推入帧队列，后续流程与软解完全一致
⚡ 特点：解码本身几乎不占 CPU，由 GPU NVDEC 完成。
💛 额外开销：av_hwframe_transfer_data() 显存→内存拷贝 (~1–3ms/帧)。
🟢 收益：1080p H.264 解码 CPU 占用降至 5–15%，降幅 ≥ 30%。

核心差异对比

维度 软解 CPU 软解 硬解 D3D11VA 硬解
解码执行单元 CPU 通用核心（x86 SIMD） GPU NVDEC 专用硅电路
帧输出位置 系统内存（CPU 可直接读写） 显存 VRAM（需 copy 回系统内存）
帧格式 AV_PIX_FMT_YUV420P AV_PIX_FMT_D3D11 → transfer → YUV420P
初始化 avcodec_find_decoder + avcodec_open2 + av_hwdevice_ctx_create + 绑定 hw_device_ctx
CPU 占用 (1080p) ~40–60% ~5–15% ✅
兼容性 所有编码格式 ✅ H.264 / H.265 主流格式；不支持的编码自动回退软解
驱动依赖 无 NVIDIA 驱动 (1650S 支持 D3D11VA)
后续管线 sws_scale → QPainter 同软解（transfer 之后完全一致）

UI 开关与回退策略

  1. MainWindow 菜单栏 — 添加「硬件加速 (D3D11VA)」勾选菜单项，默认 ✅ 勾上
  2. PlayerController — 持有 bool hwAccelEnabled_ 属性，开关切换时更新
  3. VideoDecodeThread::init() — 接收 bool hwEnabled 参数 → 若为 true 则尝试 D3D11VA 设备创建 → 成功则走硬解，失败则静默回退软解
  4. 回退不报错 — 硬解创建失败时输出日志 LOG_WARN，不弹窗不打断用户，正常软解播放
  5. 运行时切换 — 切换开关仅在下次 open() 文件时生效，正在播放中不执行热切换（避免重新初始化解码器）
