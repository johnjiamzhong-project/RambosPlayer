2.1 FilterPanel → PlayerController (GUI 线程)

FilterPanel 的每个滑块变化时，直接调用 PlayerController 的内联转发方法：

// filterpanel.cpp — onBrightnessChanged
float f = value / 100.0f;
ui->brightnessValue->setText(QString::number(f, 'f', 2));
player_->setBrightness(f);   // → PlayerController
FilterPanel 槽player_-> 调用
onEnabledChanged(state)setFilterEnabled(bool)
onBrightnessChanged(val)setBrightness(float)
onContrastChanged(val)setContrast(float)
onSaturationChanged(val)setSaturation(float)
onBrowseWatermark()setWatermark(QString)

2.2 PlayerController → VideoDecodeThread (内联转发)

PlayerController 的转发方法全部内联在头文件中，直接调用 videoDec_ 对应方法：

// playercontroller.h
void setBrightness(float v)  { videoDec_.setBrightness(v); }
void setContrast(float v)    { videoDec_.setContrast(v); }
void setSaturation(float v)  { videoDec_.setSaturation(v); }
void setWatermark(const QString& p) { videoDec_.setWatermark(p); }
void setFilterEnabled(bool on)  { videoDec_.setFilterEnabled(on); }
这一步仍在 GUI 线程。真正的跨线程发生在 VideoDecodeThread 内部——GUI 线程写 atomic 变量，解码线程读。

3. 跨线程参数传递 (atomic + dirtyFlag)

VideoDecodeThread 的 atomic 参数集

std::atomic<bool>  filterEnabled_{false};   // 滤镜总开关
std::atomic<bool>  filterDirty_{false};    // 参数变更标志（触发 rebuild）
std::atomic<float> brightness_{0.0f};     // 亮度偏移
std::atomic<float> contrast_{0.0f};      // 对比度偏移
std::atomic<float> saturation_{1.0f};   // 饱和度倍率
mutable QMutex watermarkMtx_;             // 保护 watermarkPath_
QString      watermarkPath_;               // 空串 = 无水印

3.1 写端 (GUI 线程, FilterPanel 触发)

GPU 线程调用 setter，写入 atomic 并置 dirty 标志：

// videodecodethread.cpp
void VideoDecodeThread::setBrightness(float v) {
    brightness_.store(v, std::memory_order_release);
    filterDirty_.store(true, std::memory_order_release);
}
void VideoDecodeThread::setWatermark(const QString& path) {
    QMutexLocker lock(&watermarkMtx_);
    if (watermarkPath_ != path) { watermarkPath_ = path; filterDirty_ = true; }
}
为什么 watermarkPath_ 用 QMutex 而不是 atomic？
QString 不是平凡类型，std::atomic<QString> 无法编译。
用 QMutex 保护读写；QMutexLocker 在 setter（GUI 线程）和 buildFilterDesc（解码线程）中各取一次锁。

3.2 读端 (解码线程, run() 循环顶)

每个解码循环的顶部检查 dirty 标志：

// videodecodethread.cpp — run() 循环
while (!aborted_) {
    // 1. 处理 flush 请求
    if (flush_.load()) { ... }

    // 2. 检测滤镜参数变更 → rebuild
    if (filterDirty_.load(std::memory_order_acquire)) {
        if (filterEnabled_.load()) {
            QString desc = buildFilterDesc();       // 拼滤镜字串
            filterGraph_.rebuild(desc);             // 在线重建滤镜链
        } else {
            filterGraph_.rebuild(QString{});        // 空字串 → 直通
        }
        filterDirty_.store(false, std::memory_order_release);
    }

    // 3. 取包 → 解码 → 滤镜 → push
    ...
}

4. 滤镜描述字串构造

buildFilterDesc() 逻辑

QString VideoDecodeThread::buildFilterDesc() const {
    QStringList parts;

    // 亮度 + 饱和度 → hue 滤镜
    float b = brightness_.load();
    float s = saturation_.load();
    parts << QString("hue=b=%1:s=%2").arg(b, 0, 'f', 2).arg(s, 0, 'f', 2);

    // 对比度 → colorbalance（同时调整 R/G/B midtones）
    float c = contrast_.load();
    if (c != 0.0f)
        parts << QString("colorbalance=rm=%1:gm=%1:bm=%1").arg(c, 0, 'f', 2);

    // 水印 → movie + overlay
    {
        QMutexLocker lock(&watermarkMtx_);
        if (!watermarkPath_.isEmpty())
            parts << QString("movie=%1[wm];[in][wm]overlay=10:10").arg(watermarkPath_);
    }

    return parts.join(',');
}

拼接示例

滑块状态生成的字串
亮度=0.10, 饱和度=1.00, 对比度=0.10 hue=b=0.10:s=1.00,colorbalance=rm=0.10:gm=0.10:bm=0.10
亮度=0.10, 对比度=0, 饱和度=1.00, 水印=logo.png hue=b=0.10:s=1.00,movie=logo.png[wm];[in][wm]overlay=10:10
全部默认（启用滤镜但无滑块调节） hue=b=0.00:s=1.00
滤镜关闭 ""（空字串 → 直通模式）

5. FilterGraph 生命周期

init
decode open → process
每帧 → rebuild
参数变更 → close
decode stop

init(width, height, pixFmt, tb, desc)

在 VideoDecodeThread::init() 末尾调用，desc 为空（初始不启用滤镜）。内部创建：

  1. avfilter_graph_alloc() — 建图
  2. avfilter_graph_create_filter("buffer") — buffersrc 入口
  3. avfilter_graph_create_filter("buffersink") — buffersink 出口
  4. avfilter_graph_parse_ptr(desc) — 解析用户滤镜字串
  5. avfilter_graph_config() — 连接整条链

passthrough 模式：desc 为空时仅保存宽高/格式，不创建图。
process() 直接 av_frame_clone 返回入帧。

process(AVFrame* in, AVFrame* out)

每帧解码后调用。out 由调用方 av_frame_alloc 预分配：

// 在 run() 中
AVFrame* filtered = av_frame_alloc();
filterGraph_.process(decodedFrame, filtered);
// filtered 现在持有滤镜处理后的帧，push 到 videoFrameQ

路径分支：

  1. desc 为空 → av_frame_clone + av_frame_move_ref
  2. desc 非空 → av_buffersrc_add_frame_flags + av_buffersink_get_frame

rebuild(newDesc)

用户拖滑块导致参数变更时调用。实现是

close();  init(width, height, pixFmt, tb, newDesc);

即销毁旧链，重建新链。开销 ~1-5ms，仅在用户交互时触发，不产生可感知的卡顿。

close()

avfilter_graph_free(&graph_) 释放整条滤镜链。析构函数也会调用。

7. 关键设计要点

滤镜重建在解码线程
重建发生在 run() 循环顶的 dirty 检测段，不在 GUI 线程。这样 process() 和 rebuild() 永远在同一线程串行执行，不存在竞态。
FilterGraph::process() 的 out 参数
调用方必须预先 av_frame_alloc() 分配帧对象。
av_buffersink_get_frame 要求传入已分配的空 AVFrame，传 nullptr 会导致崩溃。
watermarkPath_ 的 mutex
QString 不能声明为 std::atomic。
用 QMutex 保护，且标记为 mutable 以便 const 方法 buildFilterDesc() 也能加锁。
滤镜不可用时静默回退
若 avfilter_graph_parse_ptr 失败（如滤镜名不存在），init() 返回 false，rebuild() 返回 false。
此时 desc 已清空，自动回退直通模式，不影响正常播放。
