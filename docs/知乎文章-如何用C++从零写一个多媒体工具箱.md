# 如何用 C++ 从零写一个能播、能录、能推、能剪的多媒体工具箱？


这半年用业余时间写了一个 Windows 桌面应用，基于 FFmpeg + Qt，从项目脚手架开始，一路做到了播放、屏幕录制、RTMP 推流、视频无损剪辑全部闭环。这篇文章聊聊架构设计和踩过的坑。

先放一张架构全貌：

```
UI 交互层（播放/暂停/进度条/音量/全屏/菜单）
  │
播放控制器（统一调度生命周期）
  │
解复用模块（读取文件 → 分离视频流/音频流）
  │
  ├── 视频解码（H.264/H.265，D3D11VA 硬解）
  │     │
  │  视频滤镜（亮度/对比度/饱和度/水印）
  │     │
  │  视频渲染（YUV→RGB，QPainter 绘帧）
  │
  ├── 音频解码（AAC/MP3，重采样 S16 44100Hz）
  │     │
  │  音频输出（PCM 播放，维护主时钟）
  │
  └── 音视频同步（音频时钟驱动，视频跟随）

附属管线：
  推流：桌面采集 → H.264 编码 → FLV 封装 → RTMP 推流
  剪辑：时间轴选段 → 缩略图预览 → 关键帧裁剪 → 无损导出
```

14 个线程，3 条管线，全部通过有界阻塞队列通信，不依赖 Qt 信号槽做跨线程数据传递。

---

## 一、为什么音频要做主时钟？

这是播放器同步策略的第一个关键决策。

人耳对声音中断的敏感度远高于眼睛对帧延迟的敏感度。音频通过硬件 DMA 以固定速率播放，天然是稳定的时间基准。所以设计上让音频驱动一个原子时钟变量，视频每帧渲染前读取这个时钟，计算自己该等多久：

- **视频 PTS 领先音频 → 等待**（等音频追上再显示这一帧）
- **视频 PTS 落后音频超过 400ms → 丢帧**（追不上了，直接跳过）

这个策略简单但有效，实测同步误差控制在 2ms 以内。

## 二、为什么要自己写有界阻塞队列？

一开始想过直接用 `std::queue + std::mutex + std::condition_variable`，但很快发现三个需求不是标准库能直接满足的：

1. **push 满时阻塞生产者** —— 解码比渲染快得多，不解限的话内存很快就爆了
2. **pop 空时超时等待** —— 消费者不能死等，需要带超时退出（比如 20ms 后回去检查要不要停止）
3. **abort() 全局唤醒** —— 停止播放时必须立刻解除所有等待中的线程，这个靠裸写 condition_variable 也能做，但需要额外封装

于是就写了一个模板类 `FrameQueue<T>`，把这三个需求封装干净。视频包队列容量 100，音频包 400（音频包密度高一个数量级），解码后的视频帧队列只有 15（一帧 YUV 数据好几 MB，必须限容）。

## 三、整个项目最折磨人的模块：Seek

seek 看着简单——跳到目标位置继续播——实际上涉及四个线程的状态同步，是我在这个项目里踩坑最多的地方。挑三个典型的说：

### 坑 1：旧帧残留导致画面永久卡死

向后 seek（往回拖进度条）后，音频顺利跳到目标位置播放，但画面一动不动。排查日志发现，帧队列里残留了大量目标位置**之后**的旧帧（PTS 远大于新的音频时钟），渲染器计算 `pts - audioClock` 得到大正数，判断为"视频超前"，就暂存起来等待音频追上。但音频在新位置播放，要追上旧帧需要几百秒实际时间。于是 `pendingFrame_` 永远不被消费，画面永久冻结。

**解决**：渲染器增加对称的丢弃逻辑——不仅"落后超过 0.4s"要丢，"超前超过 5s"也要丢。正常播放时视频不可能超前 5s，这个阈值只会在 seek 后触发，精准清理残留旧帧。

### 坑 2：有效包被误清导致位置偏移

seek 流程中 PlayerController 已经清了包队列，DemuxThread seek 完后又清了一次。此时队列里只有 seek 目标位置之后的有效新包。但 AudioDecodeThread 检测到 flush 信号时，队列里已经积压了 500+ 个新包（文件 I/O 比解码快太多），它一个 `clear()` 把这些有效包全丢了。结果解码起点变成了"目标位置 + 11 秒"，音乐凭空跳了一大段。

**解决**：去掉解码线程 flush 中的主动清队列，只依赖 PlayerController 和 DemuxThread 的两步清旧，极少数的竞态漏包由 PTS 过滤器兜底。

### 坑 3：H.264 精确 seek 后画面冻结 2-5 秒

快进后声音立刻播，画面却等了 5 秒才更新。原因是精确 seek 阶段丢弃了关键帧到目标位置之间的**所有**视频包（包括 IDR 帧），解码器 flush 后收到的第一个包如果是 P 帧或 B 帧，没有参考帧就持续返回 `EAGAIN`，直到遇到下一个 IDR。而 H.264 的 GOP 最长可达 10 秒。

**解决**：精确 seek 阶段只丢弃目标前的音频包，视频包从关键帧起全部放行。目标前的旧视频帧由渲染器凭 PTS 快速丢弃（比解码器等待 IDR 快得多），画面恢复时间从 2-5 秒缩短到不到 500ms。

## 四、硬件加速的一个细节陷阱

D3D11VA 硬解启用后，`avcodec_open2` 告诉你像素格式是 `AV_PIX_FMT_D3D11`。但实际帧经过 `av_hwframe_transfer_data` 从 GPU 显存转到系统内存后，格式是 `AV_PIX_FMT_NV12`——D3D11VA 的通用软解输出格式，10-bit 内容则是 P010。

如果你用 `D3D11` 格式去初始化滤镜图（libavfilter 的 buffersrc），后面送入 `NV12` 帧的时候 FFmpeg 内部做格式校验会直接崩溃。本质原因是 `avcodec_open2` 返回的 `pix_fmt` 是硬件表面的格式，不是实际你能拿到的帧格式。

**解决**：硬解场景下固定以 `NV12` 初始化滤镜图。同时记录首帧的实际格式，如果与预期不符（比如 10-bit 内容输出 P010），动态触发重建。

## 五、Qt 线程亲和性：QAudioOutput 的坑

Qt 的 `QAudioOutput` 要求创建、`start()`、`write()`、`stop()` 全部在同一个线程执行。我一开始把它放在主线程初始化，工作线程调用——直接行为异常，偶尔卡死。

改到 `AudioDecodeThread::run()` 中创建并启动，同一个线程内完成全部生命周期，退出时 delete，函数返回前兜底清理，问题解决。这是一个很经典的 Qt 多线程陷阱：带事件循环的 QObject 子类，创建它的线程就是它的事件处理线程，跨线程调用需要 `moveToThread` 或用信号槽排队，否则未定义行为。

## 六、技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 语言 | C++17 | `std::atomic`、RAII 管理 FFmpeg 资源（AVFrame/AVPacket 手动 alloc/free） |
| UI | Qt 5.14 Widgets | 成熟稳定，QPainter 渲染视频帧，QSS 做暗色主题 |
| 音视频 | FFmpeg 4.x (vcpkg) | 全格式解封装 + 全编码器解码 + libavfilter 滤镜链 |
| 硬解 | D3D11VA | Windows 原生，免额外驱动 |
| 音频输出 | QAudioOutput | 低延迟 PCM 播放，与 Qt 事件循环自然集成 |
| 构建 | CMake + MSVC 2017 | preset 统一配置，增量编译 < 5 秒 |

## 七、还没解决的问题

**水印叠加**：FFmpeg 的 movie 源滤镜在 Windows 下无法通过滤镜图字符串正确传递含盘符的路径。`avformat_open_input` 直接打开文件完全正常，说明文件存在且可读。问题出在 `avfilter_graph_parse_ptr → av_get_token → avfilter_init_str → av_set_options_string` 这四层字符串解析链路里，盘符中的冒号 `:` 无论用单层还是双层反斜杠转义都无法正穿透到 movie filter 的 filename 选项。这个等有空读一下 FFmpeg 的 `graphparser.c` 源码再回来修。

**GPU 滤镜**：目前亮度/对比度这些走的是 CPU 软滤镜（libavfilter），如果能接到 D3D11 视频处理器上做零拷贝，性能还会有明显提升。

## 八、写在最后

市面上不缺播放器，不缺录屏工具，不缺剪辑软件。但"会用"和"懂原理"之间有一条很宽的沟——中间填的就是亲手实现一遍。

从 `av_read_frame` 到 `av_interleaved_write_frame`，从解复用→解码→滤镜→渲染的播放管线，到采集→编码→封装的推流管线，再到 seek→关键帧定位→无损导出的剪辑管线——三条管线全部贯通之后，FFmpeg 的 API 地图才算真正建立起来。

如果你也在学音视频开发，强烈建议不要停留在命令行调 `ffmpeg -i input.mp4 output.mp4` 的阶段。动手写一个播放器，哪怕简陋，收获远超看十篇教程。
