第一个设计问题：滤镜放在哪里？

这决定了 FilterGraph 的生命周期、线程模型和性能影响。

A

VideoDecodeThread 内（解码线程）

每个解码帧立即过滤镜，再推入 videoFrameQ。FilterGraph 和 VideoRenderer 完全解耦。

Pros

  • VideoRenderer 无需改动
  • 滤镜在解码线程，不阻塞 GUI

Cons

  • 滤镜增加解码线程 CPU 负载
  • 需线程安全地更新滤镜参数
B

VideoRenderer 内（GUI 线程）

帧出队后、sws_scale 之前过滤镜。FilterGraph 可直接读 UI 控件值。

Pros

  • 滤镜参数同步简单，无需跨线程
  • 解码线程不受影响

Cons

  • GUI 线程做滤镜处理，可能丢帧卡 UI
  • 违反现有 VideoRenderer 轻量化设计
C

独立 FilterThread

解码帧 → FilterFrameQ → FilterThread 处理 → videoFrameQ。滤镜有自己独立的队列和线程。

Pros

  • 解码/滤镜/渲染三级流水，各不阻塞
  • 架构最清晰的解耦

Cons

  • 增加一条队列 + 一个线程，复杂度最高
  • 当前仅 3 个简单滤镜，过度工程化

推荐 A — 在解码线程内处理。滤镜参数通过 atomic 变量传递，UI 滑块只写原子量，FilterGraph 在 run() 循环中读取。简洁且符合现有帧队列架构。
